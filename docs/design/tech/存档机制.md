# 存档机制

---

## 1. 概述

《天下牌》为一款 Roguelike 单局游戏，单局时长约 20-40 分钟。存档系统需支持中途退出后恢复进度、应用被系统挂起后数据不丢失，以及跨局累积的局外成长数据持久化。

核心设计目标：
- **中途续玩**：玩家可在任意节点退出，下次继续
- **防丢失**：应用异常退出前自动保存
- **局外成长持久化**：通宝利息、解锁进度等跨局数据
- **多存档槽**：支持 3 个独立存档 + 1 个自动存档

---

## 2. 存档范围

### 2.1 需存档的数据

| 数据分类 | 内容 | 粒度 |
|---------|------|------|
| **局内状态** | 当前层数、当前节点、已访问路线 | 每个节点 |
| **战斗状态** | 玩家手牌、敌方手牌、牌堆、弃牌堆、当前回合、牌权归属 | 每回合 |
| **玩家数据** | 已招募人物列表、已购买卜辞牌、锦囊牌、事件牌 | 实时 |
| **印记数据** | 每张牌上的印记映射 | 实时 |
| **经济数据** | 当前通宝数量 | 实时 |
| **生命值** | 当前气数、天命剩余 | 实时 |
| **地图数据** | 完整地图布局（含随机种子） | 一局一次 |
| **局外数据** | 累计通关次数、累计通宝利息、已解锁印记、初始通宝加成 | 跨局持久 |

### 2.2 不需要存档的数据

| 数据 | 原因 |
|------|------|
| 动画/特效中间状态 | 读档后重新渲染 |
| 音频播放位置 | 读档后从头播放 |
| UI 临时状态（选中高亮等） | 读档后重置为默认 |
| 已过去的事件日志（>200 条） | 缓冲区裁剪 |

## 3. 存档数据结构

```typescript
interface SaveData {
  header: SaveHeader;
  runState: RunState;
  battleState: BattleState | null;
  playerData: PlayerData;
  metaProgression: MetaProgression;
}

interface SaveHeader {
  version: string;          // 存档格式版本（如 "1.0.0"）
  slotId: number;           // 槽位编号 0-2（手动）/ -1（自动）
  timestamp: number;        // 保存时间戳
  playTime: number;        // 本局累计游戏时间（秒）
  layer: number;            // 当前层数（1-12）
  nodeType: string;         // 当前节点类型
}

interface RunState {
  seed: number;             // 本局随机种子
  currentNodeId: string;    // 当前节点 ID
  currentLayer: number;     // 当前层
  visitedNodeIds: string[]; // 已访问节点列表
  mapLayout: MapNodeData[]; // 完整地图布局
  qi: number;
  destiny: number;
  tongbao: number;
}

interface BattleState {
  playerHand: CardData[];
  opponentHand: CardData[];
  deck: CardData[];
  discardPile: CardData[];
  currentTurn: number;
  activePlayer: "player" | "enemy";
  lastPlayedCards: CardData[] | null;
  opponent: EnemyData;
  consecutivePassCount: number;  // 连续放弃次数（司马懿·隐忍等）
}

interface PlayerData {
  characters: CharacterData[];
  divinationCards: DivinationCardData[];
  brocadeCards: BrocadeCardData[];
  eventCards: EventCardData[];
  seals: SealMapping[];  // [{cardId, sealName}]
  usedOnceFlags: string[];  // 已消耗的"每场一次"技能标识
}

interface MetaProgression {
  totalClears: number;
  totalTongbaoEarned: number;
  interestTongbao: number;
  unlockedSeals: string[];
  destinyBonus: number;
  initialTongbaoBonus: number;
  deckUpgradeLevel: number;
  totalPlayTime: number;
}
```

### 牌数据最小化

牌数据仅存储最小必要字段，花色和点数可在加载时重建：

```typescript
interface CardData {
  id: string;       // 唯一标识（如 "H7"=♥7, "BJ"=小王, "RJ"=大王）
  suit: "S" | "H" | "C" | "D" | "BJ" | "RJ";
  rank: number;     // 面值 3-13, 15=ace, 20=2
}
```

## 4. 存档时机

### 4.1 自动存档触发点

| 触发时机 | 说明 |
|---------|------|
| 进入新节点 | 节点切换前自动存档 |
| 战斗结束 | 战斗结算后自动存档 |
| 黄金台消费完成 | 购买/招募后自动存档 |
| 事件牌结算后 | 事件效果执行完毕后 |
| 应用进入后台 | `visibilitychange` / Cordova `pause` 事件 |
| 应用异常 | `window.onerror` / `unhandledrejection` 回调中尝试保存 |

### 4.2 手动存档

- 暂停菜单中的"保存并退出"按钮
- 写入当前选中的手动槽位

### 4.3 防抖策略

- 自动存档两次写入间隔 ≥ 3 秒，避免频繁 I/O
- 应用进入后台时不防抖，立即写入
- 手动存档不防抖

---

## 5. 存档槽位

| 槽位 | ID | 说明 |
|------|----|------|
| 自动存档 | `-1` | 每次自动保存覆盖，不展示在槽位列表 |
| 手动槽 1 | `0` | 玩家主动保存 |
| 手动槽 2 | `1` | 玩家主动保存 |
| 手动槽 3 | `2` | 玩家主动保存 |

- 主菜单"继续游戏"优先加载自动存档
- "读取存档"列表展示 3 个手动槽，显示层数、人物数、通宝数、时间
- 新建游戏时若选中已有数据的槽位，弹出覆盖确认

## 6. 存储后端

### 6.1 优先级策略

```
IndexedDB（优先）
  ↓ 不支持/容量满
localStorage（降级）
  ↓ 不支持
内存（回退，仅当前会话）
```

### 6.2 IndexedDB（主存储）

- 数据库名：`UnderTheHeaven`
- 对象仓库：`saves`，主键 `slotId`
- 单存档预计大小 < 100KB，远低于 IndexedDB 配额
- 写入失败时降级到 localStorage

### 6.3 localStorage（降级存储）

- Key 格式：`uth_save_{slotId}`
- JSON 序列化后存储
- 注意 5MB 限制（约可存 50+ 个存档，充足）

### 6.4 Cordova 原生存储（可选增强）

- 使用 `cordova-plugin-file` 将存档写入应用私有目录
- 提供额外的持久性保障（localStorage 可能被系统清理）
- 优先级：原生文件 > IndexedDB > localStorage

## 7. 序列化与反序列化

### 7.1 序列化流程

```
SaveData 对象
  → JSON.stringify()
  → 可选：LZ-String 压缩（存档 > 50KB 时启用）
  → 写入存储后端
```

### 7.2 反序列化流程

```
从存储后端读取
  → 可选：LZ-String 解压
  → JSON.parse()
  → 版本迁移（如需要）
  → 数据完整性校验
  → 重建运行时对象（Phaser GameObjects 等）
```

### 7.3 数据完整性校验

```typescript
function validateSaveData(data: unknown): SaveData | null {
  // 1. 结构校验：必要字段是否存在
  if (!data || typeof data !== "object") return null;
  const save = data as SaveData;

  // 2. 版本兼容：版本号是否可处理
  if (!isVersionCompatible(save.header.version)) return null;

  // 3. 合法性校验：数值范围
  if (save.runState.qi < 0 || save.runState.destiny < 0) return null;
  if (save.runState.currentLayer < 1 || save.runState.currentLayer > 12) return null;

  // 4. 卡牌数据校验：54 张牌唯一性
  const allCardIds = getAllCardIds(save);
  if (new Set(allCardIds).size !== 54) return null;

  return save;
}
```

校验失败时：
- 自动存档损坏 → 丢弃，从最近的手动存档恢复（或重新开始）
- 手动存档损坏 → 标记为"损坏"，提示玩家删除

## 8. 版本迁移

### 8.1 版本号规范

格式：`major.minor.patch`（遵循语义化版本）

- **major**：存档结构不兼容变更（旧存档无法读取）
- **minor**：新增字段，可向后兼容（旧存档缺失字段用默认值填充）
- **patch**：bug 修复，不影响结构

### 8.2 迁移函数链

```typescript
const migrations: Record<string, MigrationFn> = {
  "1.0.0": noop,
  "1.1.0": migrate_1_0_to_1_1,
  "2.0.0": migrate_1_x_to_2_0,
};

function migrateSave(save: SaveData): SaveData {
  let current = save;
  const versions = sortedVersions(Object.keys(migrations));
  for (const ver of versions) {
    if (compareVersions(current.header.version, ver) < 0) {
      current = migrations[ver](current);
      current.header.version = ver;
    }
  }
  return current;
}
```

## 9. 读档流程

```
主菜单 "继续游戏" / "读取存档"
  ↓
从存储后端加载 SaveData
  ↓
版本迁移
  ↓
完整性校验 → 失败 → 提示"存档损坏"
  ↓ 通过
重建游戏状态：
  1. 还原地图布局 → 生成 MapScene
  2. 还原玩家数据（人物、道具、印记、通宝、天命）
  3. 若 battleState ≠ null → 进入 BattleScene 并恢复到当前回合
  4. 若非战斗节点 → 进入对应节点场景（黄金台/事件/地图选择）
  ↓
emit(RUN_START, { fromSave: true })
```

### 9.1 战斗中的读档特例

读档时若处于战斗中途：
- 恢复双方手牌、牌堆、弃牌堆
- 恢复当前回合数和牌权归属
- 上次出牌的牌型展示在出牌区
- 等待玩家继续操作

### 9.2 不可读档的场景

以下节点类型**不可存档/读档**（或读档后回退到节点入口）：
- 动画播放中（出牌动画、伤害飘字等）
- 锦囊牌使用确认中

实现方式：这些场景不触发自动存档，手动存档按钮置灰。

---

## 10. 局外成长数据

局外成长数据（MetaProgression）与存档槽位独立存储：

- 存储 Key：`uth_meta`
- 不受存档槽位覆盖影响
- 在任意存档通关/结束时更新
- 在主菜单读取展示（累计通关次数、可用利息通宝等）

---

## 11. 存档 UI 展示信息

存档槽位展示摘要：

| 信息 | 来源 |
|------|------|
| 当前层数 / 总层数 | `runState.currentLayer / 12` |
| 当前节点类型 | `header.nodeType`（战斗/黄金台/事件/精英/Boss） |
| 持有通宝 | `runState.tongbao` |
| 天命剩余 | `runState.destiny` |
| 已招募人物数 | `playerData.characters.length` |
| 游戏时间 | `header.playTime`（格式化为 HH:MM:SS） |
| 保存时间 | `header.timestamp`（格式化为日期） |
