# 技能系统重构计划

## 1. 当前技能实现的问题分析

### 1.1 技能逻辑散落在 GameScene 各处

当前所有技能逻辑以"检测角色是否存在 + 内联处理"的方式散落在 `GameScene.ts` 的不同方法中：

| 技能 | 位置 | 方法 |
|------|------|------|
| 韩信「点兵」 | `playDamageSettlement()` Stage 2 | 倍数亮出后 inline 实现动画+数值 |
| 文天祥「丹心」 | `playDamageSettlement()` L2220-2222 | inline 伤害加成 |
| 南蛮军「藤甲」 | `playDamageSettlement()` L2205-2208 | inline 伤害计算调整 |
| 李时珍「本草」 | `executePlay()` L1653-1671 | 打出牌后 inline 实现动画+回血 |
| 牛皋「猛攻」 | `applyPostDamageEffects()` L2479-2484 | inline 弃牌 |
| 强盗「剪径」 | `applyPostDamageEffects()` L2486-2493 | inline 偷牌 |
| 黄巾军「黄天」 | `applyEnemyGainTurnEffects()` L2496-2521 | inline 弃牌换牌 |

此外，`AIBrain.scorePlay()` L136-141 中硬编码了对南蛮军藤甲的特殊 AI 行为适配。

### 1.2 核心问题

1. **违背开闭原则**：每新增一个技能，就要修改 `playDamageSettlement()`、`executePlay()`、`applyPostDamageEffects()` 等核心方法，GameScene 不断膨胀。

2. **缺乏统一抽象**：没有"技能"这一概念的统一接口。触发时机、条件判断、效果执行三者混在一起，由方法内部的 `if` 分支筛选。

3. **动画与逻辑耦合**：金色边框、角色移动、卡牌缩放等动画代码与技能数值计算写在同一个代码块中，难以复用和测试。

4. **技能触发检查用硬编码的角色 ID 检测**：`this.playerCharacterIds.includes('hanxin')`、`this.battle.player.characterId === 'lishizhen'`、`enemyCharId === 'nanmanjun'` 等检测散落各处，新增角色需要扫描全文件找到所有检测点。

5. **多技能时序难以管理**：当前通过 `Set<PlayerCharacterId>` 追踪状态 + 内联 `delayedCall` 控制时序，未来多个技能同时触发时时序编排会非常复杂。

6. **缺乏可测试性**：技能逻辑内嵌在 GameScene 中，依赖 Phaser 运行时，无法进行单元测试。

7. **AI 行为适配耦合**：`AIBrain.scorePlay()` 中硬编码了南蛮军藤甲的 AI 逻辑，不符合"技能封装在技能自身内部"的原则。

---

## 2. 重构目标

1. **技能定义独立于 GameScene**：每个技能是一个独立类/模块，包含自身的时机注册、条件判断、效果执行。

2. **严格封装**：禁止在其它代码位置通过检测某个角色是否存在（如 `if (charId === 'hanxin')`）来实现该角色的技能效果。技能逻辑必须封装在技能类自身内部。

3. **保持「点兵」和「本草」效果和动画体验不变**：数值计算、动画时序、视觉特效保持当前水平。

4. **其他技能改为空实现（placeholder）**：保留技能类骨架，效果方法体为空或返回原值，确保编译通过且不影响游戏流程。

5. **公共机制提取**：金色边框动画、卡牌缩放、回血显示、音效播放等提取为可复用的工具方法。

6. **时序管理可扩展**：支持多技能同时触发，提供清晰的并发/串行执行控制。

## 3. 新的技能系统架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                     GameScene                           │
│                                                         │
│  ┌──────────────────┐    ┌──────────────────────┐      │
│  │  SkillEventBus   │◄───│  各个"时机钩子"调用点  │      │
│  │  (事件总线)       │    │  (在关键流程节点 emit) │      │
│  └──────┬───────────┘    └──────────────────────┘      │
│         │                                               │
│         │ 事件分发                                       │
│         ▼                                               │
│  ┌──────────────────┐                                   │
│  │ SkillRegistry    │─── 管理所有注册的技能              │
│  │ (技能注册表)      │                                   │
│  └──────┬───────────┘                                   │
│         │                                               │
│         ▼                                               │
│  ┌──────────────────┐    ┌────────────────────────┐    │
│  │  SkillExecutor   │───▶│   SkillVisualManager   │    │
│  │  (技能执行器)     │    │   (视觉特效管理器)       │    │
│  └──────────────────┘    └────────────────────────┘    │
│                                                         │
│  ┌──────────────────────────────────────────────────┐  │
│  │              各个技能类 (Skill 子类)               │  │
│  │  HanxinDianBing / LiShizhenBenCao / ...          │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 3.2 关键设计原则：按在场角色动态注册

**核心原则**：只注册当前对局中角色栏存在（或敌方角色）的角色的技能。角色栏没有的角色，其技能根本不会出现在 SkillRegistry 中。

```
对局开始 → 确定角色阵容 [playerCharacters..., enemyCharacter?]
       │
       ▼
遍历所有技能定义 → 只注册 characterId 在场角色中的技能
       │
       ▼
SkillRegistry 中仅包含活跃角色的技能
```

这保证了：
- **性能最优** — 事件触发时只遍历实际在场的角色的技能 filter
- **filter 职责单一** — 只检查技能自身的触发条件（花色、牌数等），不再需要检查角色是否存在
- **语义自然** — "角色不在场 = 技能没注册 = 不会触发"

### 3.3 数据流

```
游戏流程节点 ──► SkillEventBus.emit('时机', context) 
    │
    ▼
SkillRegistry 查到注册在该时机下的技能（仅活跃角色）
    │
    ▼
SkillRunner 按优先级顺序执行每个技能：
  1. 调用 skill.filter(context) → 检查条件（不再含角色存在检查）
  2. 条件满足 → 调用 skill.execute(context, visuals)
  3. 条件不满足 → 跳过
    │
    ▼
SkillVisualManager 提供公共动画接口（glow / card scale / heal number / ...）
```

### 4.2 事件总线

```typescript
// === src/skills/SkillEventBus.ts ===

/**
 * 技能事件总线 —— 连接游戏流程与技能系统
 * 在 GameScene 的关键方法中调用 emit() 发出事件
 */
export class SkillEventBus {
  private listeners: Map<SkillTiming, SkillContextHandler[]> = new Map();

  /** 注册事件处理器 */
  on(timing: SkillTiming, handler: SkillContextHandler): void {
    if (!this.listeners.has(timing)) {
      this.listeners.set(timing, []);
    }
    this.listeners.get(timing)!.push(handler);
  }

  /** 触发事件 */
  emit(timing: SkillTiming, context: SkillContext): void {
    const handlers = this.listeners.get(timing);
    if (!handlers) return;
    for (const handler of handlers) {
      handler(context);
    }
  }

  /** 清除所有监听 */
  clear(): void {
    this.listeners.clear();
  }
}

type SkillContextHandler = (context: SkillContext) => void;
```

### 4.3 技能注册表

```typescript
// === src/skills/SkillRegistry.ts ===

export class SkillRegistry {
  private skills: SkillDefinition[] = [];

  /** 注册一个技能 */
  register(skill: SkillDefinition): void {
    this.skills.push(skill);
  }

  /** 批量注册 */
  registerAll(skills: SkillDefinition[]): void {
    for (const skill of skills) {
      this.register(skill);
    }
  }

  /**
   * 仅注册与当前对局中活跃角色关联的技能
   * @param allSkills 所有可用技能定义
   * @param playerCharacterIds 玩家当前角色栏中的角色 ID
   * @param enemyCharacterId 当前敌方角色 ID（如有）
   */
  registerForActiveCharacters(
    allSkills: SkillDefinition[],
    playerCharacterIds: PlayerCharacterId[],
    enemyCharacterId?: EnemyCharacterId,
  ): void {
    this.registerAll(allSkills.filter(s =>
      playerCharacterIds.includes(s.characterId as PlayerCharacterId) ||
      (enemyCharacterId && s.characterId === enemyCharacterId)
    ));
  }

  /** 获取指定时机下的所有技能（按优先级排序） */
  getSkillsByTiming(timing: SkillTiming): SkillDefinition[] {
    return this.skills
      .filter(s => s.timing === timing)
      .sort((a, b) => (a.priority ?? 100) - (b.priority ?? 100));
  }

  /** 获取某个角色关联的所有技能 */
  getSkillsByCharacter(characterId: string): SkillDefinition[] {
    return this.skills.filter(s => s.characterId === characterId);
  }

  /** 清除所有注册 */
  clear(): void {
    this.skills = [];
  }
}
```

### 4.4 技能执行器

```typescript
// === src/skills/SkillExecutor.ts ===
// （为避免与接口名冲突，实际文件名/类名可用 SkillRunner）

export class SkillRunner {
  private registry: SkillRegistry;
  private eventBus: SkillEventBus;
  private visuals: SkillVisualManager;

  constructor(
    registry: SkillRegistry,
    eventBus: SkillEventBus,
    visuals: SkillVisualManager,
  ) {
    this.registry = registry;
    this.eventBus = eventBus;
    this.visuals = visuals;

    // 将注册表中的技能绑定到事件总线
    this.bindAllSkills();
  }

  /** 将注册表中所有技能的时机绑定到事件总线 */
  private bindAllSkills(): void {
    const allTimings = Object.values(SkillTiming);
    for (const timing of allTimings) {
      const skills = this.registry.getSkillsByTiming(timing);
      if (skills.length > 0) {
        this.eventBus.on(timing, (context) => {
          this.executeTiming(timing, context);
        });
      }
    }
  }

  /** 按顺序执行某个时机下的所有技能 */
  private executeTiming(timing: SkillTiming, context: SkillContext): void {
    const skills = this.registry.getSkillsByTiming(timing);
    for (const skill of skills) {
      if (skill.filter(context)) {
        skill.execute(context, this.visuals);
      }
    }
  }
}
```

### 5.2 李时珍「本草」—— 完整迁移

```typescript
// === src/skills/LiShizhenBenCao.ts ===

export const LiShizhenBenCao: SkillDefinition = {
  id: 'lishizhen_bencao',
  name: '本草',
  description: '打出牌时，回复等同于本次打出的所有梅花牌分数之和的气数',
  characterId: 'lishizhen',
  timing: SkillTiming.ON_PLAY,
  priority: 10,

  filter: (ctx: SkillContext): boolean => {
    if (!ctx.playedCards) return false;
    // 角色在场性由注册时保证
    const clubCards = ctx.playedCards.filter(c => c.suit === 'club');
    return clubCards.length > 0;
  },

  execute: (ctx: SkillContext, visuals: SkillVisualManager): void => {
    const scene = visuals.getScene();
    if (!ctx.playedCards) return;

    const clubCards = ctx.playedCards.filter(c => c.suit === 'club');
    const clubRankSum = clubCards.reduce((s, c) => s + c.rank, 0);

    visuals.startGlow('lishizhen');

    // 100ms 后放大梅花牌 + 回血
    scene.time.delayedCall(100, () => {
      const clubContainers = (ctx.centerCardContainers || [])
        .filter(c => c.getData('suit') as string === 'club');
      if (clubContainers.length > 0) {
        visuals.animateCardScale(clubContainers);
      }
      visuals.showHeal('player', clubRankSum);
      // 修改 battle 状态中的血量
      const player = ctx.battle.player;
      player.vitality = Math.min(player.vitalityMax, player.vitality + clubRankSum);
    });

    // 600ms 后停止动画
    scene.time.delayedCall(600, () => {
      visuals.stopGlow('lishizhen');
    });
  },
};
```

## 7. 时序管理方案（多技能同时触发）

### 7.1 问题描述

当前技能按硬编码顺序执行（先李时珍、再韩信、再牛皋等），未来可能出现以下场景：
- 多个技能注册在同一时机（如 `ON_COEFFICIENT_REVEALED` 上有韩信「点兵」和南蛮军「藤甲」）
- 其中一个技能包含动画，需要等待动画完成

### 7.2 方案：优先级 + 串行队列 + 并发动画

```
SkillRunner.executeTiming(timing, context)
  │
  ├─ 获取该 timing 下所有技能（按 priority 排序）
  │
  ├─ 依次执行每个技能的 filter
  │     ├─ filter 返回 true → 加入待执行队列
  │     └─ filter 返回 false → 跳过
  │
  └─ 按队列顺序执行：
        ├─ 技能 A.filter → 技能 A.execute（可能包含动画）
        ├─ 等待技能 A 的动画完成（通过 Promise/回调）
        ├─ 技能 B.filter → 技能 B.execute
        └─ ……
```

### 7.3 实现方案：SkillRunner 支持异步

```typescript
// === src/skills/SkillRunner.ts ===

export class SkillRunner {
  // ……

  /**
   * 异步执行某个时机下的所有技能
   * 技能按优先级串行执行，但单个技能内部的动画与下一个技能的
   * filter/execute 之间可以通过返回 Promise 来同步等待
   */
  async executeTimingAsync(
    timing: SkillTiming,
    context: SkillContext,
  ): Promise<void> {
    const skills = this.registry.getSkillsByTiming(timing);
    for (const skill of skills) {
      if (skill.filter(context)) {
        const result = skill.execute(context, this.visuals);
        // 如果执行器返回 Promise，等待完成后再继续下一个技能
        if (result instanceof Promise) {
          await result;
        }
      }
    }
  }
}
```

## 11. 关键风险与注意事项

1. **时序同步风险**：`executePlay()` 中李时珍技能使用 `delayedCall(600)` 等待动画完成后调用 `continueAfterSkill()`。迁移后需确保通过 `await skillRunner.executeTimingAsync()` 保持相同等待时间（600ms），否则后续流程会提前触发导致动画不完整。

2. **系数文字引用**：韩信技能需要修改 Stage 2 创建的 `coeffText`，必须确保该引用在技能执行时仍然有效（未被销毁）。

3. **伤害信息的可变性**：韩信技能会修改 `damageInfo.coefficient` 和 `damageInfo.finalDamage`，南蛮军藤甲也会修改 `damageInfo.sumRanks`。两者通过 `priority` 控制执行顺序（藤甲 priority=5 → 韩信 priority=10）。

4. **AI 南蛮军适配**：`AIBrain.scorePlay()` 中对 nanmanjun 的特殊评分逻辑不应直接删除，而应通过 `SkillTiming.ON_AI_SCORE` 事件让南蛮军藤甲技能进行评分调整。placeholder 阶段可先将旧逻辑保留（因为藤甲技能 filter 返回 false 不会执行），待藤甲技能完整实现后再移除。

5. **向后兼容**：Phase 1-2 不删除任何现有代码，仅新增基础设施。Phase 3 开始逐步替换。

6. **测试覆盖**：因项目暂无自动化测试，每阶段完成后需手动验证：（a）韩信对战的点兵动画+系数计算；（b）李时珍对战的本草动画+回血数值；（c）其他角色对战的基础流程不报错。
